# OpenComputer v2 Memory Merge Layer — Phase 2 Best-Practices Brief

## 1. Context

v2 today has the *substrate* of a strong memory system but no layer that unifies it on the agent recall path. Concretely, what exists:

- **Session FTS5** — local, sub-millisecond keyword recall over conversation history (`agent/memory_manager.py`, `memory-stack/recall_probe.py` already gating it at `RECALL_FLOOR=0.8`).
- **A dormant holographic store** — SQLite + FTS5 + HRR fact store (`plugins/memory/holographic/store.py`) with `add_fact` (content-dedup), `search_facts` (ranked `fact_id` list), `update_fact`, `record_feedback`. Substrate present, not on the recall path.
- **Honcho** — the identity/representation provider, occupying v2's single external-provider slot.
- **GBrain** — a Postgres-native knowledge brain reachable over MCP (entities, relations, hybrid+graph retrieval).

The gap: `MemoryManager.build` today **concatenates** each provider's pre-formatted text block in order (builtin first, one external), with **no fusion, no ranking, no timeout, no dedup** across stores [memory_manager.py]. A hung provider stalls the turn; a weak vector hit can outrank a strong graph fact; the same fact can live in two stores and drift. The missing piece is a **retrieve-and-merge layer** that sits above these four stores, fans out, fuses, ranks, bounds latency, and renders one clean context block. That layer is what Phase 2 designs.

## 2. What the best teams do (cross-cutting patterns)

Five patterns recur across every serious system surveyed (Mem0, Zep/Graphiti, Letta, Anthropic, GBrain, Honcho, the production search engines):

1. **Parallel fan-out, then rank-fuse.** Fire every store concurrently, keep each store's native scoring, and combine only at the merge step via **Reciprocal Rank Fusion (RRF)**. RRF uses rank not raw score, so incompatible scales (BM25 unbounded vs cosine 0..1 vs graph-walk vs Honcho relevance) never get compared directly. k=60 is the universal default [Elastic RRF; LangChain EnsembleRetriever; GBrain RETRIEVAL.md; Azure AI Search].

2. **Hybrid recall + a real rerank stage, never cosine alone.** The leaders combine semantic + lexical + graph, then rerank a small candidate window with a cross-encoder. Anthropic measured retrieval failures dropping 5.7% → 3.7% (contextual embeddings) → 2.9% (+BM25) → 1.9% (+rerank); GBrain's graph arm alone is +31 P@5 points [Anthropic Contextual Retrieval; GBrain RETRIEVAL.md; Zep].

3. **Extract-and-reconcile on write, never blind append.** The write path is an LLM-controlled state machine: extract salient facts, retrieve similar existing ones, emit ADD/UPDATE/DELETE/NOOP. This is what keeps a store from rotting into contradictory near-duplicates [Mem0 two-phase pipeline].

4. **Two-tier memory + invalidate-don't-delete.** A small always-in-context tier (self-curated durable facts) over a large retrieved tier; and superseded facts are marked invalid with bi-temporal validity rather than deleted, so "as of" reasoning and audit trails survive [Letta core/archival blocks; Zep four-timestamp edge invalidation].

5. **Background ("sleep-time") formation off the hot path + bounded latency on read.** Expensive extraction/consolidation runs after the turn; reads enforce per-store timeouts and fuse partial results so the slowest store can never stall the turn [LangMem hot-path-vs-background; InfoQ hedged requests; Mem0 async writes].

6. **Two eval layers: deterministic retrieval metrics every commit, LLM-judge QA periodically — both stratified by question type.** Recall@k / MRR / nDCG over a frozen gold set as the cheap CI gate; LLM-as-judge accuracy (a *different* judge model, averaged over runs) nightly [LoCoMo; LongMemEval; BEIR; v2 `skills/agent-eval`].

## 3. Decisions we adopt for v2

**Architecture: a thin MergeLayer above the four stores, edge not core.**
Replace `MemoryManager.build`'s concat-providers step with: gather ranked candidates from each provider, fuse, optionally rerank, render one ranked block. No new core tool, no store changes. *Rationale: fits v2's "capability at the edges" idiom; the stores stay decoupled* [memory_manager.py; v2 AGENTS.md].

**Combination strategy: parallel-query-then-RRF-fuse — NOT routing-first, NOT pure layered.**
For each turn, fan out to all four planes concurrently, take each plane's ranked list, and fuse with RRF (`score = Σ 1/(k+rank)`, k=60). *Rationale: the three remote planes answer overlapping-but-different questions and we rarely know in advance which holds the answer; routing-first silently drops the store that had it (a wrong route is an unobservable recall miss), and pure layered/cascade under-escalates when the cheap layer looks confident but is wrong* [Elastic RRF; LangChain; query-routing failure-mode analysis]. **Hybrid refinement:** keep local FTS5 as an always-on synchronous backbone, and apply a cheap rules/LLM-tool router *only* to gate the expensive remote calls (Honcho/GBrain), always with a parallel fallback so a misroute cannot zero out recall.

**Heterogeneous unification + ranking.**
Each provider returns **candidates**, not pre-baked text: `{id, text_for_rerank, source_store, native_rank, native_score?, metadata}`, via a per-store adapter (FTS5 row → text; Honcho peer-rep string → text; GBrain node/triple → `subject — relation — object`). Fusion operates on `id + native_rank` only, so it is type-agnostic; rerank operates on `text_for_rerank`. Rank heterogeneous results in three composable stages: (1) **weighted RRF** with a per-plane weight multiplier (default 1.0) so identity/Honcho can be up-weighted on personal-fact queries; (2) a **source-tier multiplicative prior** applied as an outer re-rank (user-authored local markdown and curated GBrain pages > bulk auto-captured chat/Honcho rows; demote stale/archive at 0.5×, never hard-filter; bypass the prior for temporal queries so fresh context resurfaces); (3) an optional **cross-encoder rerank** on the top ~20–30 fused candidates only. Add **per-source floors** so a sole-source identity fact (only Honcho can hold it) is not buried by consensus boosting. *Rationale: RRF sidesteps the impossible cross-scale normalization; the tier prior + floors fix RRF's two known weaknesses (it discards magnitude and demotes unique-source answers)* [GBrain source-aware CASE ranking; Azure per-retriever weights; Hindsight; cross-store consensus-demotion analysis].

**Latency / timeout strategy.**
Local FTS5 is the synchronous backbone and always completes. Each **remote** plane (Honcho, GBrain) gets a **hard per-store timeout set from its measured p99**; on deadline, fuse whatever returned and drop laggards — RRF degrades gracefully because a missing store contributes 0. Keep the cross-encoder reranker and any LLM query-expansion **default-OFF**, gated behind a cost tier, on only for the synthesis path where a downstream LLM hides the latency. *Rationale: in a fan-out, p99 tracks the slowest leaf; without a deadline one wedged provider blows the whole turn budget — exactly v2's current risk* [InfoQ hedged requests ~74% p99 cut; Pinecone two-stage; Anthropic 150→20 prune].

**Write path: extract-and-reconcile, one plane per fact, dedup, no double-writes.**
Adopt Mem0's two-phase write as the v2 write contract, implemented in `memory_manager.py` + the provider plugins (not a core tool), run on a **capable** model even when recall uses Haiku: extract salient candidates from (rolling summary + recent window + new turn), then per candidate retrieve top-k similar and emit ADD/UPDATE/DELETE/NOOP. **Route each write to exactly one plane by type** — this is the single rule that prevents drift:
- **Honcho = identity/representation only.** Write raw conversation turns + ingested events (stripping `tool_use` blocks), never pre-digested facts; let the Deriver produce the representation. Bootstrap the peer card only at known-identity moments (signup/correction), remembering `set_card` **replaces** the whole 40-fact card [Honcho README/representation.mdx/peer-card.mdx].
- **GBrain = entities + relations** (and the self-wiring zero-LLM link graph on write).
- **Local markdown/holographic FTS5 = exact, verbatim, immediately-recallable facts** (paths, keys, IDs, decisions).

Dedup at write time via entity resolution + recency-wins; mark superseded facts invalid (bi-temporal `t_valid`/`t_invalid` + `t_created`/`t_expired`), never delete — extend the existing `agent/memory_versioning.py`. Run extraction **in the background** (the existing dreaming plugin), keeping the hot path fast. *Rationale: one-plane-per-fact + reconcile is what stops the "same fact in two stores → drift" failure v2 has already been bitten by; bi-temporal invalidation gives correct "as of" reasoning without a full graph DB* [Mem0; Zep; Honcho clean-seam; Hindsight four-lever; v2 memory_versioning.py].

**Per-agent isolation + opt-in promotion.**
Keep **isolated-by-default** — v2 already runs subagents with `skip_memory=True` and the parent sees only the delegate summary [delegate_tool.py]. Bridge via the **existing `on_delegation` hook** (`agent/memory_provider.py`; called at `delegate_tool.py:2638`): have the active provider optionally write a **distilled** record into the orchestrator's store, gated by a `config.yaml` flag `memory.promote_subagent_results: {off(default) | explicit | summarize | auto}`. Promote only the **condensed summary** (already `entry['summary']`), run it through the existing injection-threat scan, extract atomic facts, dedup with recency-wins-and-invalidate, and write into a single explicit `orchestrator/shared` namespace that only the orchestrator prefetches. For heavy outputs, promote an **artifact reference**, not the content (v2 already has the artifact registry + `artifact.created` SSE). *Rationale: isolation protects focus/privacy; CrewAI-style auto-share-everything is the documented production failure ("no per-user isolation, fail fast"); gated promotion through the existing seam needs zero new core surface* [Anthropic orchestrator-worker; blackboard paper; CrewAI; v2 on_delegation seam].

**Eval design: fixed gold set + two metric layers + CI gate.**
Build `tests/tools/test_memory_recall_eval.py` (or a `skills/memory-eval/` mirroring `agent-eval`) driving `holographic.store.MemoryStore` directly — no new core tool. Freeze a repo-local **gold set** as YAML: ~30–50 facts (`corpus`) + ~20 `queries` each carrying BEIR-style `relevant_fact_ids`, spanning **single-hop / multi-hop / temporal / knowledge-update / abstention** (abstention = empty relevant set). Compute deterministic **Recall@{1,3,5,10}, Precision@k, Hit-Rate@k, MRR, nDCG@10** as the **per-commit CI gate** (fail if Recall@5 or MRR drops below a frozen floor, mirroring `recall_probe.py`'s `RECALL_FLOOR=0.8`); extend it to **cross-store fused recall@5**, not just FTS5. Add **write-quality cases**: right-fact-stored, dedup (same `id`), knowledge-update (new value must **outrank** old, not merely be present), abstention (empty/low-trust, no hallucination). Gate **p95 search latency** and **tokens-injected-per-recall** as named budgets too. Layer an LLM-judge QA eval on a **nightly/release** cadence using a **judge model different from production**, averaged over ≥5 runs, results **stratified by question type**. Pin the gold-set **query style** to the FTS5 keyword retriever (or test the NL→keyword gap explicitly) — fold `recall_probe.py`'s cases in so the FTS5 implicit-AND failure (0.62 NL vs 1.00 OR) cannot silently return. Add a RULER-style **scale-stress** case (plant 1 fact among 10/100/1000 distractors, assert recall@5 holds). Wire into CI exactly like `agent-eval`: `eval.py --threshold … || exit 1`, with `--json` output. *Rationale: deterministic retrieval metrics are the cheap diffable regression gate; LLM-judge is the expensive periodic truth signal; both are needed and neither alone is sufficient* [LoCoMo; LongMemEval; BEIR; Mem0 BEAM; Zep/Mem0 dispute on run-averaging; v2 recall_probe.py + agent-eval].

## 4. What we explicitly avoid

- **No raw-score blending across stores.** BM25 vs cosine vs graph-walk vs Honcho relevance are incomparable; any fixed normalization is brittle and lets one store dominate by scale accident. RRF only. Min-max-weighted fusion is an opt-in upgrade *after* we can calibrate (it breaks on single-result stores and is outlier-sensitive) [OpenSearch normalization caveats].
- **No routing-away from the local store, and no unbounded fan-out.** Only the expensive remote planes are routed/gated, always with a parallel fallback; every remote call has a deadline and partial-result fusion [tail-latency analysis].
- **No double-storing facts across planes.** Never push structured/exact facts into Honcho expecting a fact KB, and never call `peer.chat()` for verbatim recall — it synthesizes. One plane per fact, by type [Honcho failure modes].
- **No always-on cross-encoder or LLM query-expansion on cheap/high-volume paths** — real per-query latency and token money; gate behind cost tiers and keep them vendor-pluggable (local bge-reranker / FlashRank / API) per v2's model-agnostic rule [Anthropic; Pinecone; v2 model-agnostic standing rule].
- **No auto-applied contradiction/temporal edits.** A probe that auto-writes `valid_until` silently rewrites legitimate change-over-time; emit paste-ready resolution commands, gate writes behind one explicit phase, and distinguish temporal evolution from real contradiction [GBrain contradictions.md "never auto-applies"].
- **No blind append, no cheap model on the write/reconcile path, no store without a decay/archival sweep** — these are the three classic store-rot failures; reconcile on write, use a capable write model, reuse v2's `.archive/` curator for bounded growth [Mem0; Hindsight; Letta].
- **No default-ON promotion, no broadcasting subagent memory, no promoting raw child transcripts** — isolated-by-default, opt-in only, condensed summaries only [CrewAI/AutoGen failure modes].
- **No caching merged results on the query string alone, and no new always-loaded core memory tool** — fold the full retrieval config (query + plane weights + embedding model/provider + mode knobs) into the cache key, versioned; keep all merge logic at the edge [GBrain knobs_hash; v2 AGENTS.md].
- **Do not bypass the existing `sanitize_context` / injection-threat fence** — fuse *after* sanitization so untrusted store output cannot smuggle prompt-injection into the ranked block [v2 build_memory_context_block threat-scan].

## 5. Open design questions for Phase 3

1. **Router boundary:** what exactly triggers gating a remote plane (rules vs a cheap LLM tool-call vs semantic classifier), and what is the measured wrong-gate rate we will tolerate before falling back to always-parallel?
2. **Per-plane RRF weights and the source-tier map:** ship sensible defaults (all 1.0) or learn them from the gold set? Which query-intent classes (identity/temporal/entity/exact) flip which weights?
3. **Timeout budgets:** what are Honcho's and GBrain's *actual measured* p99s on the live stack, and therefore the real deadline numbers (we must measure, not guess, or we starve the richer-but-slower graph plane on every turn)?
4. **Reranker default + provider:** local bge-reranker/FlashRank vs an OpenRouter/Cohere endpoint — which is the default seam, and is it ever on outside the synthesis path?
5. **Bi-temporal scope:** which fact classes get full bi-temporal validity in `memory_versioning.py` vs simple recency-wins, and where does the temporal-verdict enum live (six-member, per GBrain) to avoid crying wolf on every metric that legitimately moved?
6. **Promotion default for the explore-agents personas:** stay `off`, or do the preset personas (Atlas/Forge/etc.) warrant `summarize` by default given they each already own a memory scope?
7. **Honcho read-your-writes:** how does the eval harness handle Honcho's async-derivation lag (poll `queue_status` / keyword-search fallback) so cross-store fused-recall tests are not flaky?
8. **Markdown-as-system-of-record commitment:** do we adopt GBrain's full FS-canonical + reconciler + byte-identical-round-trip + CI-gate discipline for the local plane, or a lighter version — and what is the disaster-recovery rebuild guarantee we promise?
