## 1. What dreaming does TODAY

The dreamer is a faithful v1 port that distils durable, user-specific facts from recent session history and graduates the high-signal ones into the always-injected `MEMORY.md`. The whole pass runs through one entry point, `runner.py:94 run_dream_cycle`:

- **Pass 1 (re-score):** `_rescore_dreams` (runner.py:290) re-grades the `DREAMS.md` holding pen so held facts graduate as recall accumulates.
- **Pass 2 (ingest):** `candidates.build_session_digests` (candidates.py:64) reads RAW user/assistant turns from `state.db.messages` READ-ONLY (mode=ro, candidates.py:57), noise-filters each turn (`_is_noise_line`, candidates.py:194), and builds one ~4000-char digest per session. `llm.extract_facts` (llm.py:119) emits third-person "The user..." facts; they are exact-deduped, then collapsed by `cluster.cluster_candidates` (greedy single-linkage cosine >=0.7, cluster.py:26).
- **Three-gate pipeline** `DreamingPipeline.run_once` (engine.py:214): SCORE (`llm.score_fact` >=0.65), RECALL (`count_sessions_matching` FTS5 distinct-session count >=2, candidates.py:263), DIVERSITY (cosine to nearest `MEMORY.md` entry <0.8). Routing (engine.py:327): all pass -> PROMOTE (`memory_io.promote`, dated "(dreamed YYYY-MM-DD)" marker, capped 20/run); diversity-pass-but-score/recall-fail -> HOLD in `DREAMS.md`; diversity-fail -> DROP, or SUPERSEDE in place when `llm.decide_supersede` returns UPDATE (engine.py:401).
- **Live couplings that work:** SENSE->DREAM is live via `outcome_link.adjusted_score_threshold` (outcome_link.py:30), which nudges the promotion bar +/-0.05 from recent `turn_scores` (relax on good outcomes, tighten on poor). The anti-recursion invariant `is_derived_fact` (candidates.py:311) excludes any provenance-tagged "(dreamed ...)" line from re-extraction (model-collapse guard). `dream_orchestrator.run_all` (dream_orchestrator/__init__.py:75) drives three DreamTarget adapters (local/honcho/gbrain) and the one-way `honcho->gbrain->local` cross-feed (`importer.run_cross_feed`, importer.py:167), promoting verbatim via `memory_io.promote_raw`. `self_evolution/cycle.run_cycle` (cycle.py:131) sequences SENSE->DREAM->ENRICH->EVOLVE nightly at `0 4 * * *`.

## 2. The amplification thesis (one paragraph)

The dreamer is the ONLY component that holds both the prose fact AND the FTS5 ground-truth of which tokens actually resurface for that fact, yet it throws all of that evidence away and appends opaque prose to a frozen system-prompt snapshot. The single highest-leverage move is therefore to stop discarding the dreamer's own exhaust: route every promoted fact (with its already-computed recall count, score, outcome band, and OR-expanded keywords) out-of-band into the holographic FTS5 facts table the MergeLayer is being built to rank over, via the injectable `promote_fn` slot in `_build_pipeline` (runner.py:254) with ZERO engine change. That one act converts dreaming from a write-only markdown mutation into the populating writer of the second retrieval plane, and it is the exact Phase 3 Decision C step. Everything else (semantic dedup, observability, self-scheduling the federation) either feeds that writer cleaner data or makes its output visible and trustworthy.

## 4. Top 3 BUILD-NOW in detail

### BUILD-NOW #1: Dual-write promoted facts into the holographic FTS5 facts table

**Exact seam.** `runner.py:254 _build_pipeline` is the injectable-callable factory; the engine (`engine.py:193`) is pure and host-injected, so a new writer plugs in as the `promote_fn` with ZERO `engine.py` change. The fact already carries the frozen marker shape from `is_derived_fact`/`_DERIVED_MARKER_RE` (candidates.py:311/305), so the dual-written row is excluded from re-dreaming automatically.

**How it interlinks.** This is Phase 3 Decision C ("route exact/verbatim facts to holographic FTS5 out-of-band", PHASE3:307-401, write-path seams at :460-481 naming `plugins/dreaming/`). It gives the MergeLayer its second plane to fuse over (`agent/memory_manager.py:473-493 prefetch_all -> MergeLayer.recall()`), and it is the compaction/retention target req-9 connects curator+dreaming to (PHASE3:558). The `source_store='dreamed'` tag is what the supervisor and Dream Report later read.

**Independently testable + reversible.** Testable in isolation: feed a synthetic promoted fact through `_build_pipeline` with the new writer injected, assert one row lands in the holographic facts table with the right content, provenance marker, and tag, and assert the SAME fact is excluded by `is_derived_fact` on the next pass (anti-recursion). No server, no MergeLayer needed. Reversible: the writer is an injected `promote_fn`; remove the injection and dreaming reverts to markdown-only. The holographic rows are provenance-tagged so they can be bulk-deleted by `source_store='dreamed'`.

### BUILD-NOW #2: Canonical-keyword + per-fact trust sidecar at promotion
